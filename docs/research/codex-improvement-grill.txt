**Verdict**

Build tool retrieval first. Current hot path sends every schema on every model call: [loop.ts](/Users/aktanazat/Documents/projects/personal/maestro/src/agent/loop.ts:145) calls `registry.toolSpecs()`, and [registry.ts](/Users/aktanazat/Documents/projects/personal/maestro/src/tools/registry.ts:112) only has “all specs” mode. Literature says narrow tool exposure improves cost and selection. This is not replacing registry. It is retrieval-gating over registry.
**Top 5 By Benefit / Effort**
1. **Dynamic tool retrieval layer** — first build. M effort. Highest direct benefit: prevents free-tier token limit, cuts tool-schema prompt tokens likely 50-80%, improves tool choice.
2. **Gate-failure CRITIC / Reflexion repair memo**. M. Run only after failed gate or repeated same tool error; store concise failure lesson in ledger/mission log.
3. **Agentless-style localization → repair → validation path**. M. Strengthen `code.localize_failure` and add `code.related_files` / import graph ranking.
4. **Query-aware context compression / evidence memory**. M. Upgrade [context.ts](/Users/aktanazat/Documents/projects/personal/maestro/src/agent/context.ts:117), not full MemGPT cosplay.
5. **ADaPT-style adaptive subagent decomposition**. S/M. Spawn only when uncertainty/failure threshold is hit; don’t make every plan recursive.
